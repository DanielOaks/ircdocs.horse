Internet Relay Chat specifications

Lots of behaviour in the IRC protocol as implemented in clients and servers today is not formally specified. If you are looking to implement IRC software, the best thing to do is look at specifications, information, and how other software behaves as a guide.

Useful Specifications

RFC1459

The canonical IRC specification. At this point, this spec is very old. However, it's still one of the primary references for most software out there.

It's recommended to implement IRC based off this specification, finding real-world differences by comparing your software with other clients and servers around.

RFC7194

6697 is IRC's TLS port. It's now in an RFC. Hooray.

This also specifies some details about certificates that are useful to keep in mind.

RPL_ISUPPORT Hardy Internet-Draft

Every server implements this. This is mandatory for any client implementing any sort of intelligence. Yes, this is an Internet-Draft and not an RFC. Ignore that. Even though this is not an IETF standard, RPL_ISUPPORT is implemented by everything.

Regarding specific RPL_ISUPPORT keys used these days, look at various servers and networks.

IRCv3 Working Group

This is a working group made up of a whole bunch of IRC software authors. IRCv3 specifications are written by various software authors, submitted, and accepted if the majority of the group agrees on them. These are recommended and tend to introduce new and useful features to the IRC protocol.

Regarding specific IRCv3 specifications, here are rough support matrixes for clients and servers.

Not-As-Useful Specifications

RFC2812

An update to RFC1459. This update, however, was not universally accepted. Do not take this specification too seriously. Most software doesn't give this spec much sway, and consider it a suggestion if anything.

It's generally best to work off RFC1459 and find implementation differences from there, given the amount of content in RFC2812 that's been outright rejected and not implemented.
